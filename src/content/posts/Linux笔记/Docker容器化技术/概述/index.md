## Docker架构与容器化
### Docker架构分为三个部分
- Client：命令行
- Docker Host：安装了Docker的主机
- Registry：类似应用市场

### 基本操作流程

#### 如果你想要直接拉取市场中现成的程序
在Docker Host中使用`docker pull redis(程序名)`来**拉取**这个程序 包括本体和环境 之后使用`docker run redis`来运行这个程序 使用`docker run`运行的程序就叫作**容器** 

#### 如果你想要上传自己的程序和配置
在Docker Host中使用`docker build xxx(程序名)`先发送一个构建请求 之后使用`docker push xxx(程序名)`去**推送**你的程序 之后在别的Docker Host上 使用`docker push xxx` `docker run xxx`就可以启动容器了

## 什么是容器？
我们先来看传统的部署架构
分为三层
- 应用
- OS
- 硬件
这样的**三层**架构有一个缺点：应用之间不隔离 如果一个应用发生了问题(比如内存泄露) 会影响到其它应用的运行 从而导致环境直接炸了

再来看看虚拟化部署
分为四层
- 应用
- Hypervisor
- OS
- 硬件
这个**四层**架构解决了传统部署架构的痛点：它有多个独立的虚拟OS 每个应用都可以部署在自己独立的虚拟OS内 一个应用炸了不会影响到其它应用
但是这样的架构有一个缺点：每一个应用都有自己的OS 就会导致整个系统非常笨重 部署时间也非常长

最后就来看看容器部署
应用的容器包含了一个应用所需的全部的环境 而且容器之间互相隔离 并且非常轻量 部署时间极短

### 容器的特点
- 容器类似轻量级的VM
- 容器共享操作系统内核
- 容器拥有自己的文件系统、CPU、内存、进程空间等
- 容器互相隔离

### 容器部署的优势
- 轻量
- 快速
- 隔离
- 跨平台
- 高密度
- etc
