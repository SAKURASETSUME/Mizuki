**由于http协议传输数据本质上是明文传输 即使使用了对称加密算法来加密明文 那只要攻击者知道了对称加密算法的加密算法 就能解出明文 再比如你自定义了一个加密算法并加密明文 那服务器端也需要对应的key来解密 这个key怎么传输又是一个问题 陷入了循环 所以说http协议是不安全的**
## Https底层算法

**非对称加密算法 -> 客户端有公钥 服务器端有私钥 公钥加密 私钥能解密 私钥加密 公钥能解密 公钥加密 公钥不能解密公钥加的密 但是这样也并非完全安全 如果攻击者伪造成了服务器 把用户的请求拦截住 给了用户假的公钥 当用户发送数据时 数据实际上传输到了攻击者的服务器 攻击者再使用假的私钥来解开用户的密文 之后加工成对自己有利的数据 再用真正的服务器返回的真正的公钥加密发送给真正的服务器 这样就实现了身份的伪造**

## CA机构参与保证互联网安全

**整个流程为：服务器需要先向CA机构提交公钥 CA机构会验证提交者是否是服务器的管理员(比如通过让提交者在服务器创建一个文件 CA机构去访问那个域名查看是否能访问通) 认证通过后 CA机构会把服务器的公钥用CA机构自己的私钥加密 最后以证书的形式下发给服务器 当服务器向用户发送数据 还有用户向服务器发送数据的时候 操作系统内置的CA机构的公钥会对证书进行解密 查看证书是否是服务器申请过的证书 如果中间被攻击者篡改了 那么认证就不通过 数据无法进行传输(攻击者是可以解开证书的 但是没有CA机构的私钥 无法发送假数据还有篡改数据)**
